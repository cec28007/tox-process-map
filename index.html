Identifier Model

Five identifiers come together on each biocomp-type study. Understanding how they relate is central to LIMS modeling.

IdentifierIssued byIssued whenPurpose
Quote # / PO #Sales (Salesforce)Customer signs quote + issues POCommercial anchor
Product Code (a.k.a. Test Code)Pre-defined catalog (TX0xxx)Each line item on the quoteIdentifies which test (GPMT, MEM elution, pyrogen, etc.)
GLP NumberTest Facility ManagementWhen study is assigned on the GLP Assignment ListIdentifies the study as GLP; may not be needed long-term
Sample ID = Test Article #Tally (homegrown, all ROC labs)At receipt of physical test articleTracks the physical material; sample and test article are synonymous
Animal IDAnimal Tracks (homegrown)At animal receipt or assignment to a testPer-individual animal traceability for in-vivo tests — carries weights, rest periods, study history

Supporting Detail

Service Buckets

Three standardized service types plus one placeholder bucket for custom, non-standard work.

Bucket A — BET (Bacterial Endotoxins)

  • One line item equals one test equals one result.
  • ROC pools samples; SLC runs individually.
  • Destructive; extraction per PYT-8 or method validation (QA-396 §B).
  • Simple case for LIMS.

Bucket B — Cytotox (MEM Elution / Direct Contact)

  • Structurally same as BET for intake and billing.
  • Used as screening and as a line item inside biocomp packages.
  • Extraction per QA-396 §B (Elution) or §C (Direct Contact).

Bucket C — Biocompatibility (the complex one)

Typical "Big 5" tests in a biocomp package:

  1. Intracutaneous Irritation
  2. Systemic Injection
  3. Guinea Pig Maximization Test (GPMT)
  4. MEM Elution / Cytotox
  5. Material-Mediated Pyrogen

Resource sharing. ~13 devices typical: 6 for the top three tests (share polar + non-polar × 3 time points), 1 for MEM, 5 for pyrogen, 1 spare.

But each test still has its own product code, protocol, invoice line, and report.

Bucket D — Complex / Custom Studies [PLACEHOLDER]

  • 8–10 per year (P&G, PopBio, Idera, etc.).
  • Heavy change-order traffic; milestone-based billing already in use.
  • Out of scope for v1 detail — shown as a placeholder lane on the map; process to be captured later.

Billing Trigger Design

Current pain: flat 50% up-front creates about $200K stuck when test articles never arrive (2025 biocomp backlog). Proposed target state: four event-driven milestones.

MilestoneEventTrigger sourceRationale
M1Protocol completed & customer-signedLIMS protocol status = SignedReal work delivered; avoids stuck-article problem
M2Test article received or animal order placedTally Sample ID issuedMaterial cost commitment
M3Launch (first dose)LIMS study status = In-life startedAnimal AP hits net-30; ~10-day gap from animal invoice to launch is the cash-flow concern
M4Final report deliveredLIMS report status = ReleasedCloseout

Supporting Detail

LIMS Modeling Implications

First-cut observations to feed the LIMS data model for Toxicology.

  1. Shared test-article pool per study. LIMS needs a test-article inventory concept where one receipt feeds multiple product-code tests. Lindsey World calls this the S-number → child-sample tree; the paper analog today is TAD sheet QA-2.
  2. Time-point-based sub-sampling. Extracts are recreated at t=0 / 14 / 21 — not aliquoted from a frozen stock. LIMS must model "extract creation event" as a distinct record per time point.
  3. Animal traceability bridge. In-vivo endpoints must link to Animal Tracks records. [?] Animal Tracks is homegrown; an API or export path should be feasible.
  4. Environmental monitoring bridge. Rees probe data should be pullable into the study record (today it is fetched manually). [?] Rees API / export format TBD.
  5. IACUC protocol link. Each study carries an IACUC protocol number separate from the study protocol. [?] Confirm whether captured as a field today.
  6. Endpoints vary by protocol (confirmed by Donna). Endpoint schema must be protocol-driven, not product-code-driven. Templated and study-specific forms both exist today.
  7. Complex studies. Keep as a separate study type with a minimal data model; do not force them into the standardized flow in v1.
